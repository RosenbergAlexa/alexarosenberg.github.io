## Enhancement One

The artifact which I selected for the project as a whole was the Project 2 program from IT-145 – Foundation in Application Development. To originally create this artifact, students were challenged to create an application for the company Grazioso Salvare, a search and rescue animal training company. The company required an application which could help track search and rescue animals, and students were to contribute to the completion of this task. The final solution allowed users to run the application, interact with the program through the console, add service dogs or monkeys, update their status, reserve an animal, and see the whole list of available animals.
Since this artifact comes from an introductory class, it is no surprise that it is a very simplistic example of what a software solution could be. This is not the fault of the code or system design, but rather a reflection of the material being taught to students at that time. It does, however, mean that the artifact as a whole has significant potential in terms of possible improvements, and it is for this reason that I selected it as my project artifact. In its original state, the artifact only utilizes the console to interact with users, lacks security and data persistence measures, and is tightly coupled in design.

To showcase my skills and abilities in software development, my priority when approaching upgrade of this system was to target high value updates. Or rather, which system updates would provide the greatest value to the end user. And the answer to that question pointed to the user interface and lack of security measures. In software systems which need to interact with a user, the usability, look, and feel of the presented user interface is one of the first things that is judged. If the user interface doesn’t provide users with the functionality they need while still looking nice, then users will reject the overall product for something else. This makes having a good UI a high priority item for software systems. Additionally, if the system cannot provide a measure of security, then it does not matter how good the UI is.
The first enhancements to the original artifact which I implemented were to replace the console UI with a modern desktop UI, and to implement authentication and authorization security measures. 
Implementation of the UI and security enhancements was planned to improve the ability of the system to support a number of possible users. Since in an organization, all members have their own role and responsibilities, any software intended to support a group of people needs a way to ensure that all users gain access to only the functionality they need. A wholistic approach to solving this issue involves making sure that the system not only has a functional user interface which supports moderating access to the system and a user’s functionality access but also requires the security implementation to ensure that users can identify themselves so that their allowed level of access can be defined and remembered.

In addition, the implementation of these enhancements was planned to ensure that all communication to the user through the user interface was professional, clear, and communicated the necessary information without being overly verbose. If the implemented user interface cannot communicate the necessary information to the user without causing confusion, then the UI cannot be considered successful. Since the intention is for an organization with potentially many users utilizing the application to meet the needs of the company, the enhanced UI had to present information in such a way that it was possible for users to understand how to use the software successfully.

Implementation of a modern UI is not a trivial task, especially in this era where applications can live or die based solely on their appearance rather than their functionality. This turns the goal of implementing a UI into a task not only requiring no small amount of planning, knowledge, and skill in programming, but also a measure of basic design and artistic skill as well. There are many ways to implement a modern-feeling user interface, including using built-in Java libraries, using HTML and CSS, or using third-party frameworks to create windowed applications. In assessing the available options, the timeline was probably my most serious consideration given that this enhancement included just more than a UI. Previous experience with the Java UI libraries left an impression of them being clunky and out-of-date, while CSS and HTML could be used the timeline was too short to be able to create something polished to the look and feel that I desired the enhanced artifact to possess. This led to my selection of a third-party framework to make a windowed interface, which I felt was possible to achieve with the timeline I had given I have previous experience with implementing these types of interfaces.

I did want to challenge myself with this enhancement, so I selected a UI framework with which I had no experience with, JavaFX. JavaFX is an open-source Java framework for the creation and delivery of desktop and rich web applications. The creation of interface windows with JavaFX uses FXML, an HTML-like markup language that is used to define all the individual components of a JavaFX window. FXML files can be written by hand or can be created using a free window design application SceneBuilder. Given my lack of experience with FXML and my short timelines, SceneBuilder ended up being a valuable tool in not only creating windows but also being able to visualize what users might see and interact without having to put in a significant amount of effort to set up visualization tests.

Another particular benefit of the selection of JavaFX as the UI framework was that the framework inherently supports modern software architectural design patterns like MVC and MVVM. Since another stated outcome of complete enhancement of my selected artifact was to reduce the level of coupling in the code, one of the other challenges I posed for myself was to follow the MVC architectural pattern while implementing the UI and security enhancements. The MVC pattern separates software into three components: the model, the view, and the controller. Each component takes responsibility for a particular application responsibility, enforcing a separation of concerns and ensuring that while each piece can work the others to create a functional solution, each can be created, tested, and maintained separately. Enforcement of the MVC architecture pattern was a deliberate choice to not only improve the overall maintainability of the final enhanced artifact but also showcase a level of consideration for software design and development best practices.
My lack of experience with JavaFX was a significant challenge for this overall enhancement. Particularly because the security enhancements could not really be implemented without a functioning user interface. The first security enhancement which could be implemented was the user authentication, which is non-operable without having a user login window. Thus, my design choices and self-challenges really forced me to learn on the fly in order to complete everything which I had planned to accomplish with this enhancement. Once I had a functional user login interface and got more comfortable with how the JavaFX toolkit supports the MVC design, implementation of the other planned security enhancement – user authorization – was much more straightforward.

I feel that I met these desired outcomes throughout the implementation of the security and user interface enhancements. I will state that final judgement of the usability of a system’s UI should not be left to the original developer, since they have more intimate knowledge of the system’s functionality behind the user interface.

I found that throughout the implementation of the UI and security, that the complexity of the artifact significantly increased. It wasn’t unexpected, since the original artifact was exceedingly simple, and most systems requiring modern user interfaces follow design architecture patterns made to make the systems more manageable to maintain. Given the nature of a UI with security enhancements, I did find there to be a challenge with testing with standard unit testing. JavaFX applications can be unit-tested using a separate framework such as TestFX or Monocle to test the UI in headless mode. This allows individual components of the application to be run without requiring interaction with the view, so the business logic can be tested separately. At this point of development of the application, there is very little business logic to speak of and most of the functionality was wrapped up in the interface itself. Given this and the timelines I had, I ultimately prioritized testing the behavior and responses of the interface over spending time to figure out how to write unit test cases, although this is something that I would like to go back and review in the future.

## Enhancement Two

At this point in the enhancement process, the original artifact has been given a modern windowed UI and much needed security measures. Basic functionality from the original artifact was maintained in the first iteration of the enhanced version, including the ability to add new animals and reserve existing ones. Additional functionality was introduced as well, such as the ability to edit the training status of existing animals, which serves to make the enhanced artifact more comprehensive in the sense that it provides a more complete toolset for controlling the introduction and management of the rescue animals. But there is always room for improvement, and for using the opportunity to showcase additional skillsets. This leads us to enhancement two.

The proposed enhancement to the artifact to improve it beyond the original functionality was to implement a search feature. This would allow users to enter the name or animal type that they are looking for and have the user interface limit the view to only those animals matching the desired criteria. A big consideration for this proposed enhancement was what was the most efficient way this goal could be achieved, and what sorts of data structures could be utilized to ensure the goal was met.

If we review the original state of the artifact, before enhancement one, the most complex data structure utilized was a dynamic array that was used to keep track of a collection of custom objects. Now given the functionality scope and limitations of the original artifact, this is not necessarily a bad design decision. Arrays are efficient when attempting to access data and do keep elements sorted throughout the lifecycle of the array, so the use of dynamic arrays adds security against out-of-bounds exceptions and attacks. However, there is little value gained in ensuring that the collection of rescue animals remain sorted throughout the lifecycle of the application, meaning that the only true benefit to the use of arrays in the original artifact is the ability to quickly access items when needed.

Unfortunately, arrays and dynamic arrays are not efficient data structures when it comes to searches. With O(n) potential comparisons being required to find any given animal in the collection, the amount of time it takes the application to find and retrieve an animal is inversely proportional in efficiency to the number of animals being managed. Since users are provided with the ability to change the reservation status of an animal, the more animals known, the slower the system will become due to the inefficiency of searches and the dynamic array data structure. While it is possible to improve search algorithm efficiency with dynamic arrays, however, it requires that the collection of data be pre-sorted.

The main limiter to sorting rescue animal data is the lack of a clear sorting order. There are many pieces of information used to define a single animal, meaning there could be a significant number of combinations of critical information used to sort the data in any sortable data structure. Likewise, it could also be argued that an unsorted data structure which is already optimized for additions and retrievals – like the hash table – be utilized instead. This would be a critical efficiency improvement to the original artifact, as the hash table has an efficiency of O(1), making the application much faster with regards to finding a specific item. The argument against the use of hash tables is that the system must know the identity of the item which it needs to retrieve. In the scenario of the rescue animal inventory where the system needs to find the first available match rather than knowing the identity of the animal needed to begin with, the hash table may be no more efficient than a standard dynamic array. This issue could potentially be mitigated by changing the key used to store information in the hash table, but again the issue with what information is critical and strictly unique about each individual animal arises and creates a blocker.

Of course, any implementation design considerations must be made with consideration to the already existing UI. There is little point in a search feature if it does not result in a change to the appearance or displayed information in the view displayed for users. This made the JavaFX framework the biggest design limiter to this enhancement. JavaFX provides a significant benefit in terms of an enhanced and modern UI desktop application, however whatever data structure chosen to hold animal information had to be compatible with what JavaFX can handle. Failing to implement a solution compatible with the already existing UI implementation would result in a custom search solution which may be even less efficient than just using the data structures available through JavaFX.

Which leads into the next point, how exactly can an efficient filter and search function be implemented in a UI using the JavaFX framework? The answer lies in the data structures which JavaFX makes available for storing collections of data. The base data structure, ObservableList, is a concrete implementation of the Java List interface, making them akin to Java’s ArrayLists. Use of the ObservableList was already integrated into the enhanced artifact from enhancement one. More specifically, since rescue animals are displayed as a list of items in the inventory window, the collection of rescue animal objects representing the data for each of the animal cards in the list is held in a JavaFX ObservableList. This allows the UI controllers to create the UI components correctly, display the animal data, and allow users to edit individual animals while keeping the entire collection accessible as users move between windows.

With a search function, you want the ability for the UI to remove items from the displayed list of animals without deleting any of the information to do so. In this sense, we want to be able to apply a custom filter to the ObservableList and have the UI window automatically update to reflect the application of a filter to the list of animals. To do this with JavaFX, you must wrap your original ObservableList in a FilteredList. While this might sound complicated, all it translated to was creating a new FilteredList variable using the ObservableList. Once the FilteredList is created and available, filter conditions can be applied based on the text that the user enters in the UI’s search field. The FilteredList also has the benefit of having listeners which observe the filter and the underlying data contained in the ObservableList, thus any modifications to the data such as new items being added or modifications being applied are propagated instantly and dynamically evaluated by the filter.

The unfortunate downside about using FilteredLists is that because they are a required data structure provided by the JavaFX framework, there is no way to evaluate or improve the efficiency of the underlying algorithm. So, in order to implement a dynamic and responsive UI, you must use a FilteredList, whose underlying search algorithm cannot be optimized or modified in any way. I did spend a bit of time researching ways to improve the performance of the FilteredList, but most of the information that I found pointed to the applied filter and the number of items in the ObservableList. As far as optimizations go, the key is to keep the filter conditions as simple as possible. In the case of the enhanced artifact, that meant keeping the search limited to animal type or animal name. Setting the search feature to just those data fields is limiting and means that users cannot search based on other useful information like animal breed, age, or even reservation status. Because of this, the search feature may be something which I revisit later to see how its functionality can be expanded without sacrificing overall performance of the application.

One of the major things that I noticed once the implementation of the search algorithm was completed was that while a filter could be applied, the only apparent structure or order. Every animal was displayed in the order that they were originally placed in the ObservableList, which essentially gives them a sort of order-entered-into-the-system. It wasn’t necessarily a bad way to display information about the rescue animals, but I figured it could be better, so I challenged myself to do some more research to figure out how to apply a sort order to the data that didn’t entirely depend on offloading that presorting of data to an integrated database. The result of this research led me to SortedLists, yet another JavaFX provided collection data type specifically provided for applying a sort to data meant to be utilized in UI features.

Implementation of the SortedList is extremely similar to how FilteredLists are implemented, but in this case since we want the filtered search data to also be sorted, we wrap the FilteredList in a SortedList. Once you do that, you can define the sort conditions. In my case, the sort was based off animal name, with a sub-sort of animal type applied just in case there were animals sharing the same name. The same negatives that applied to Filteredlists also apply to SortedLists. There is no way to improve the sort algorithm, and improvements in performance especially those which require re-sorts are dependent upon the complexity of the sorting conditions. 
Overall, implementation of this enhancement was successful, although it did require a significant amount of research into the types of collection data types that JavaFX could use to display collection information in a window. As mentioned earlier, there are some areas where the implemented functionality could be improved, and it would be very valuable to go back and implement unit tests in order to be able to isolate tests for the search and sort functionality.

## Enhancement Three

At this point in the enhancement plan, the original artifact has been updated to have a modern desktop user interface, updated security including authentication and authorization measures, and additional application functionality including new edit and search features. If we applied no further enhancements to the application, it would remain as what could best be described as a UI showcase piece. All currently implemented functionality centers around user interaction with the interface, which is intentional as that is how users will interact with the application, however even the enhanced artifact lacks data persistence measures which mean it has lacks long-term memory with regards to interactions with users. 

Unfortunately, that does mean several inherently negative things about the state of the enhanced artifact. One, that the identity of what could arguably be critical company information which should be reasonably well protected is hard coded into the application directly and can therefore be accessed by reverse engineering the application. Two, that since the system cannot remember the results of any interaction with its data from any user, it can – and should – be argued that it fails to achieve its main purpose of helping an organization manage its collection of rescue animals. And three, that the application does not support collaboration in the sense that each instance of the application installed on independent computers is an island of knowledge to itself and does not communicate modifications to system inventory with other application instances. Furthermore, the security mechanisms implemented during enhancement one are also arguably functionally defunct as well since user credentials and authorization levels are hard-coded and cannot be modified.

This leads us to the last planned enhancement of the artifact: integration of a database. By moving all the application data to a database, we can implement a practical solution to the issues identified in the state of the enhanced artifact post enhancement two. The issues with application security caused by hardcoded user authentication and authorization data are mitigated, and more collaborative control of application access is enabled given that user access can be modified or revoked without required redeployment of the entire program. With modifications to application data being recorded in the database when they are performed by users, we also solve the issue of the application having only short-term memory. This also serves to increase the collaborative nature of the enhanced application by making it possible for multiple instances of the application to exist and interact with a single data source rather than utilizing their own private data sets.

The question then becomes what kind of database to use and where to host it. Use of a NoSQL database like MongoDb typically fits less structured data or scenarios where huge datasets are expected. It is quite a valid approach to use a NoSQL database for this integration since we know that certain animals are represented by different information than others in the inventory, and this use case fits quite well. Arguably, however, this application doesn’t expect to be handling huge data sets, and as such, should not be considered a major selection feature when considering what database type should be selected. SQL databases, by comparison, are relational databases where data is organized and stored in tables where data is categorized by columns and records have data relating to each column of the table it is entered into. In the selected artifact, data is heavily relational for both users and animals being tracked, and it is arguable that a relational database schema is more understandable from a data modeling perspective than a NoSQL schema would be. A major benefit of SQL databases is that most commercially available distributions are ACID compliant, which ensures that data remains valid and uncorrupted even in the event of power outages, system failures, or concurrent user access. This feature of being ACID compliant isn’t limited to just SQL databases however, as MongoDB is also compliant for individual documents and for sets of documents. Taken all together, both NoSQL and SQL databases have their features and drawbacks for use and implementation in this feature enhancement. Since there is no clear winner, database selection for this enhancement is something that can be left to developer preference rather than directed by clear feature requirements.

With regards to where to host the database, we have two options: host it locally to the computer of origin or host it in the cloud. Hosting it locally provides a level of security for the application, as there is a reduced risk of attack and data loss given that an attacker would have to gain access to the computer the application is installed on to get access to the database. The unfortunate downside is that shareability of data between instances of the application installed on different computers remains limited. This could be mitigated by some sort of API hosted in the cloud which uses some sort of messaging queue to receive data messages from each application and disseminate the changes to the other instances, however that drastically increases the overall complexity of the system. Hosting the database in the cloud makes the database available to all instances of the application regardless of what computer they are installed on, which solves the problem identified in the original artifact. It does come with a downside of issues with how to correctly hide database access credentials and is an issue which should be acknowledged. The more secure solution would be to host an API in the cloud to act as a middleman between the application and the database so that the credentials to the database are not hard-coded or stored in an encrypted file. Again, going the route of the API drastically increases the complexity of the overall solution, and was not something that I had time to implement.

Ultimately, I chose to implement a MySQL database remotely hosted in the cloud. The selection of MySQL over other SQL databases was a selection born purely of having experience with MySQL and therefore being comfortable with the tools available to support its creation and maintenance. Additionally, I chose to use Aiven to host the database in the cloud to make the final application better support collaboration for a company with multiple users all working together to support a collection of rescue animals.

With the selection of a database and host server, the next logical step was to design the database and implement it so that the hard-coded application data could be migrated to the database and removed from the enhanced artifact. This leads to the next and most obvious conundrum when implementing a relational database schema: what is all the information we need stored, how can it be categorized, and what collection of tables should be used to represent everything correctly? If we think about it objectively, we can define two categories of data which need to be stored, user identity and authorization data, and animal identity and status data. These two categories don’t really have a link to each other as we are not linking animals to users, but the implementation of two different databases and the subsequent juggling of swapping connections to ensure the correct database is accessed would be more of a hassle that it is worth. In the end, I chose to use a single database and have two sets of unrelated table schemas, one for user data, and the other for animal data.

Users themselves have a collection of identifying information that should be kept track of individually, such as names, email, and an encrypted hash of their password – since it is a significant security risk to directly store passwords in a database. There is also a need to keep track of the allowed roles in the application as they relate directly to the level of functionality access which should be granted in the application. Thus, there are a total of three tables which I used to represent the necessary data. The first and likely most obvious is a table to keep track of user information since that is critical data required to gain access to the application, the second is a table to keep track of roles, and the third is a permissions table which relates a user to a role. With this schema, we can ensure that data is properly stored, related, and available for query when a user attempts a login action. When a user attempts to log in, we can query the users table to retrieve the password hash for a user with the entered username. If the user is known and the password matches, the next thing to do is determine the level of authorization access they should be granted so this may be returned to the application. We can retrieve this information through a slightly more complicated query, which involves identifying the user’s role listed in the permissions table, and retrieving its associated name from the roles table.

Animals, on the other hand, have a lot more information associated with them. They also have the added complication of being represented by differing categorical information depending upon the species of animal. Because of this, we can split information representing a single animal into three categories: core data which all animals must be defined by, animal type data which may or may not be partial depending on the actual type, and status details defining training level and reservation status. Each of these data categories were translated into a table in the database schema, and both animal type and status tables were related to the main animals table using foreign keys. Thus, if we want to update the animal inventory with a new animal, we can query the database to first determine if an animal with the same animal type and name exist by cross referencing the animals table with animal types table. If we receive no matching records, then we can create a new entry in the animal types table and ask the database to give us back the record’s unique row identifier. After the type entry is created, we may add a record to the animals table with the requested animal information, matching the type details value to the new record entered in the animal types table while the connection to the correct status entry can be set using a sub-select query of the status table.

The original artifact, as mentioned previously, was never designed to support the use of standard database commands. Meaning that all considerations required when integrating use of a remote SQL database had to be considered to change the architecture of the program to accommodate this enhancement. The greatest consideration that had to be made was how to handle sending standard CRUD commands to the database given that there are two different ways they can be handled: synchronously or asynchronously. Handling everything in a synchronous manner is the easiest option as it requires no additional complexity from multithreading. However, in an environment with a dynamic UI like the one created with Enhancements One and Two, completely blocking the entire program while a database command is sent and the response is awaited before the user can do anything else can cause the program to act like it is unresponsive, significantly reducing the quality of the user experience. Given this, the option that I settled on was to use multithreading to send database commands on background threads, freeing the main UI thread to handle the user interface while the background thread pool handled communication to the remote database.

Choosing to utilize threads to handle database communication does increase the complexity of the enhanced artifact, however, since it is typically accurate to say that any scenario requiring the use of multiple threads requires a greater level of understanding of what the program may be doing at any given time. Additionally, use of standard threads is not supported by JavaFX, which was a bit of a complication in the implementation process. Thankfully, JavaFX does provide its own thread class, but it comes with its own syntax and learning curve. A valuable bit of functionality provided with JavaFX’s Task class is the ability to define what happens when the background thread hits certain execution states such as scheduled, running, completed, and failed. This ability proved very valuable, as it allows for the definition of variable UI behavior depending on the outcome of the database command sent. For example, in the case of adding a new animal entry to the database, the task handling the execution of the database commands was defined with an event handler that closes the window and returns to the main inventory window if the animal data is added to the database without error, and the response to any failures was to have the task capture the exception message and display it in the UI window for the user to respond to.

It’s one thing to discuss the complexity of implementing a multithreaded scheme to invoke calls to the database, but that discussion assumes that the implementation of a way to communicate with the database, handing all standard CRUD database commands, and parsing responses properly has already been done. This is a necessary assumption; you cannot really create a background thread to execute non-existent commands. Implementing database communication with Java isn’t too difficult, using the standard JDBC driver makes things relatively painless, beyond, of course, deciding on how best to protect the database connection credentials. To keep all the database functionality in a centralized location, I ended up writing a database service class with static methods performing the functionality the various controllers required. One of the major benefits of this design choice was to reduce the number of locations in which the database credentials could be found, and to decouple direct communication with the database with the functionality of the UI. If the database ever changes, the database service can be rewritten without demanding cascading changes in other parts of the artifact as well. 

The biggest impact that this enhancement had on the artifact is data persistence. The original artifact was unable to remember anything. Changes, additions, or modifications to animal information were immediately lost upon close of the application, basically making it inoperable from a practical usage viewpoint. Implementation of a back-end database which saves data added or modified from user interactions with the UI means that the enhanced application is now always able to “remember” what happened in previous runs. This enhancement changes the artifact from a mundane UI showcase piece to an actually functional application which can achieve its purpose of managing and maintaining an inventory of animals.
